# 4.5 Dokumentimi teknik

## 1. Pershkrim i pergjithshem i aplikacionit

ITM Exam eshte nje aplikacion web per menaxhimin e provimeve universitare. Sistemi eshte ndertuar si platforme full-stack dhe mbeshtet disa role te perdoruesve: Admin, Professor, Assistant dhe Student. Secili rol ka qasje ne funksionalitete te ndryshme sipas pergjegjesise qe ka ne procesin akademik.

Qellimi teknik i aplikacionit eshte te centralizoje proceset kryesore te provimeve: krijimin e struktures akademike, regjistrimin e studenteve, krijimin e provimeve, menaxhimin e pyetjeve, zhvillimin e provimit nga studenti, ruajtjen e pergjigjeve, evidentimin e shkeljeve, vleresimin dhe publikimin e rezultateve.

### 3.1 Backend

Backend-i ndodhet ne:

```text
backend/OnlineExam.Api
```

Pjeset kryesore:

- `Controllers/` - permban endpoint-et REST, si auth, users, exams, questions, enrollments dhe reports.
- `Models/` - permban entitetet kryesore te databazes.
- `DTOs/` - permban objektet qe dergohen/kthehen nga API.
- `Data/AppDbContext.cs` - konfigurimi kryesor i Entity Framework Core.
- `Migrations/` - historiku i ndryshimeve te skemes se databazes.
- `Program.cs` - konfigurimi i aplikacionit, JWT, CORS, Swagger, database migrations dhe seed data.

### 3.2 Frontend

Frontend-i ndodhet ne:

```text
frontend
```

Pjeset kryesore:

- `components/` - komponente te perbashketa si `AppShell`, language switcher dhe panele dashboard.
- `pages/` - faqet kryesore te aplikacionit.
- `pages/admin/` - faqe per administrim, academic structure, users dhe enrollments.
- `pages/exams/` - faqe per exam list, exam create, question bank, attempt, gradebook, monitorim dhe results.
- `lib/` - API clients si `examsApi.js`, `academicApi.js`, auth helpers dhe i18n.
- `routes/` - protected routes dhe konfigurimi i navigimit.
- `styles/` - CSS kryesor per layout, cards, forms, tables, buttons dhe responsive design.

## 5. Instalimi dhe ekzekutimi lokal

### 5.1 Kerkesat paraprake

Para nisjes se projektit duhet te jene te instaluara:

- Git
- .NET 8 SDK
- Node.js dhe npm
- PostgreSQL ose Docker Desktop
- Nje editor si Visual Studio Code

### 5.2 Klonimi i projektit

```powershell
git clone https://github.com/almamuzliukaj/Online-Exam.git
cd Online-Exam
```

### 5.3 Nisja e databazes

Nese perdoret Docker Compose:

```powershell
docker compose up -d
```

Nese perdoret PostgreSQL lokal, duhet te krijohet database dhe te vendoset connection string ne konfigurimin e backend-it.

Shembull connection string lokal:

```text
Host=localhost;Port=5432;Database=online_exam;Username=postgres;Password=your_password
```

### 5.4 Nisja e backend-it

```powershell
cd backend/OnlineExam.Api
dotnet restore
dotnet run
```

Kur backend-i starton:

- aplikon migrations;
- krijon/ruan skemen e databazes;
- mund te seed-oje demo data nese konfigurimi e lejon;
- hap API-ne per frontend.

Ne development, Swagger zakonisht hapet ne URL-ne qe shfaqet ne terminal me `/swagger`.

### 5.5 Nisja e frontend-it

Ne terminal tjeter:

```powershell
cd frontend
npm install
npm run dev
```

Frontend zakonisht hapet ne:

```text
http://localhost:5173
```

### 5.6 Environment variables

Frontend duhet te dije URL-ne e backend-it:

```text
VITE_API_BASE_URL=http://localhost:5045/api
```

Ne production, kjo vendoset ne Vercel me URL-ne publike te Railway backend.

Backend perdor konfigurime si:

```text
ConnectionStrings__DefaultConnection
Jwt__Key
Jwt__Issuer
AllowedOrigins
SeedDemoData
```

Secrets si `Jwt__Key` nuk duhet te ruhen ne GitHub.

## 6. Menyra e perdorimit te aplikacionit

### 6.1 Login

Perdoruesi hap faqen `/login`, vendos email dhe password, dhe sistemi e autentikon me JWT token. Pas login-it, perdoruesi drejtohet ne dashboard sipas rolit.

Rolet kryesore:

- Admin
- Professor
- Assistant
- Student

### 6.2 Admin workflow

Admin perdor aplikacionin per te pergatitur strukturen akademike.

Hapat kryesore:

1. Login si Admin.
2. Hap `Academic`.
3. Krijon ose kontrollon Term, Course dhe Course Offering.
4. Hap `Users`.
5. Krijon ose aktivizon studentet/profesoret/asistentet.
6. Hap `Enrollments`.
7. Zgjedh term, vit dhe semester.
8. Zgjedh studentet.
9. Krijon semester enrollments.
10. Aktivizon enrollments.
11. Gjeneron current semester courses qe studenti te behet eligible per provime.

Pa kete lidhje mes studentit dhe course offering, studenti mund te mos shoh provimet ne `Available Exams`.

### 6.3 Professor/Assistant workflow

Professor ose Assistant perdor aplikacionin per menaxhim te provimeve.

Hapat kryesore:

1. Login si Professor/Assistant.
2. Kontrollon `Assigned offerings`.
3. Hap `My exams` ose `Create exam`.
4. Zgjedh course offering.
5. Vendos titull, pershkrim, duration dhe total points.
6. Shton pyetje nga question bank ose krijon pyetje te reja.
7. Publikon provimin kur eshte gati.
8. Pas tentimeve te studenteve, hap gradebook.
9. Kontrollon pergjigjet, pike dhe notes.
10. Publikon rezultatin.

### 6.4 Student workflow

Student perdor aplikacionin per te pare dhe mbajtur provime.

Hapat kryesore:

1. Login si Student.
2. Hap `Available exams`.
3. Shikon provimet ku eshte eligible.
4. Hap provimin dhe lexon rregullat.
5. Fillon exam session.
6. Pergjigjet ne pyetje.
7. Sistemi ruan pergjigjet dhe mund te regjistroje integrity violations.
8. Studenti ben submit.
9. Pasi profesori publikon rezultatin, studenti e sheh ne `My results`.

## 7. Funksionalitetet kryesore

### 7.1 Authentication dhe Authorization

- Login me email/password.
- JWT token per request-et pas login-it.
- Role-based access per Admin, Professor, Assistant dhe Student.
- Endpoint-et backend kontrollojne rolin dhe qasjen.

### 7.2 User Management

Admin mund te krijoje, importoje, aktivizoje ose menaxhoje perdoruesit.

### 7.3 Academic Structure

Admin menaxhon Terms, Courses, Course Offerings dhe staff assignments. Keto te dhena jane baza per krijimin e provimeve dhe eligibility te studenteve.

### 7.4 Enrollment Control

Enrollment lidh studentin me semester dhe course offering. Ky proces vendos nese studenti ka te drejte te hyje ne provim.

### 7.5 Question Bank

Question bank perdoret per ruajtjen e pyetjeve, kategorizim sipas lendes/ofertimit, MCQ/text/code/SQL questions, import/gjenerim/review te pyetjeve dhe perdorim te pyetjeve ne provime.

### 7.6 Exam Creation

Professor krijon exam per assigned offering. Forma per krijim perfshin Course Offering, Title, Instructions/Description, Duration, Total Marks, Question Selection dhe Publish/Draft workflow.

### 7.7 Student Exam Session

Studenti sheh pyetjet e provimit ne exam session. Sistemi mbeshtet timer, pyetje nje nga nje, answer saving, submit, integrity warning, violation count dhe kthim ne aplikacion pas submit.

### 7.8 Integrity Events

Sistemi regjistron ngjarje te dyshimta gjate provimit, si humbje fokusi, ndrrim tab-i, dalje nga fullscreen, tentativa per copy/paste/right click dhe shkelje te tjera te exam policy.

### 7.9 Gradebook dhe Results

Professor mund te shikoje attempts, kontrolloje pergjigjet, vendose pike per pyetje, llogarise total/percentage/grade dhe publikoje rezultatet. Studenti sheh vetem rezultatet qe jane publikuar.

### 7.10 Reports dhe Export

Sistemi ka faqe raportimi me tabela, filters dhe export CSV per rezultatet, participation dhe integrity activity.

## 10. Informata shtese per kuptim te projektit

### 10.1 Pse studenti nuk e sheh provimin?

Arsyet me te zakonshme:

- studenti nuk eshte aktiv;
- studenti nuk ka semester enrollment;
- enrollment nuk eshte aktivizuar;
- nuk jane gjeneruar current semester courses;
- provimi nuk eshte publikuar;
- studenti nuk eshte eligible per course offering;
- backend/API nuk po komunikon me databazen e duhur.

### 10.2 Pse professor nuk sheh offering?

Arsyet me te zakonshme:

- professor nuk eshte assigned ne course offering;
- user role nuk eshte Professor/Assistant;
- offering nuk eshte aktiv;
- te dhenat jane ne term tjeter.

### 10.3 Pse login nuk funksionon ne deployment?

Kontrollo:

- `VITE_API_BASE_URL` ne Vercel;
- `AllowedOrigins` ne Railway;
- `Jwt__Key` ne Railway;
- connection string te PostgreSQL;
- `/health` endpoint te backend-it;
- CORS errors ne browser console.

### 10.4 Kufizime aktuale

- Browser nuk mund te bllokoje plotesisht shortcut-et e sistemit operativ.
- Integrity monitoring bazohet ne detection/logging, jo kontroll total te pajisjes.
- AI/question generation workflows kerkojne konfigurim dhe kontroll manual para perdorimit serioz.
- Deployment varet nga konfigurimi korrekt i environment variables.

## 11. Perfundim teknik

Projekti eshte nje aplikacion full-stack me strukture reale universitare. Ai perfshin frontend, backend, database, autentikim, authorization, exam workflows, question bank, enrollments, grading, results, integrity logs dhe deployment. Sistemi mund te testohet lokalisht dhe ne production nese jane konfiguruar sakte databaza, API URL, JWT secret dhe CORS origins.
